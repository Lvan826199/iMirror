# 真机联调手册

从零开始，Windows / macOS / Linux 三个系统的完整步骤。
每一步都写了**要执行的命令、预期结果、不对时看什么**，不需要懂 Python。

## 0. 你需要什么

- 一台 iPhone 或 iPad（不需要越狱，系统版本不限，越新越有验证价值）
- 一根**支持数据传输**的数据线（纯充电线不行，这是最常见的坑）
- Windows / macOS / Linux 电脑任意一台（要有装软件的权限）

## 1. 下载项目

**方式 A：用 git（装过 git 的话）**

```bash
git clone https://gitee.com/xiaozai-van-liu/imirror.git
```

**方式 B：下载 ZIP（没装 git）**

浏览器打开 https://gitee.com/xiaozai-van-liu/imirror ，
点右上角橙色「克隆/下载」→「下载 ZIP」，解压。

> Windows 建议解压到**不带中文和空格**的路径，如 `D:\imirror`；
> macOS 随意，如 `~/imirror`。

## 2. 安装环境（按你的系统看对应小节，其余跳过）

### 2.1 Windows

**第 1 步：双击运行 `scripts\setup-windows.bat`**

脚本自动完成：装 uv（自带 Python，无需单独装）→ 建虚拟环境 → 装依赖
（含 libusb，无需手动放 dll）→ 跑 22 个离线测试 → 装 ffmpeg → 环境自检。
全程几分钟，不用操作。

预期窗口结尾（此刻还没插手机/没换驱动，最后一个 ✗ 是正常的）：

```
✓ pyusb 已安装
✓ libusb 后端可用
✗ 未发现 Apple USB 设备 (vid=05ac)     ← 正常
```

常见问题：
- 双击弹出「Windows 已保护你的电脑」→ 点「更多信息」→「仍要运行」
- winget 装 ffmpeg 失败 → 不影响采集，只影响最后播放验证，可后面用 VLC 代替
- 显示「[失败] 安装中断」→ 把窗口内容完整复制反馈

**第 2 步：连接手机并信任（⚠️ 必须在换驱动之前做）**

1. 数据线连接 iPhone 和电脑
2. 解锁 iPhone → 弹出「要信任此电脑吗？」→ 点**信任**，输一次锁屏密码
3. 没弹？拔掉重插一次，保持手机解锁

> 顺序很重要：信任弹窗依赖苹果原始驱动，要趁驱动还没换的时候完成。

**第 3 步：换驱动（唯一需要手动点的环节，一次性）**

1. `Win + R` → 输入 `services.msc` 回车 → 找到 **Apple Mobile Device Service**
   → 右键**停止** → 双击它，启动类型改为**禁用**
   （列表里没有这个服务？说明没装过 iTunes，直接进入下一小步）
2. 浏览器下载 [Zadig](https://zadig.akeo.ie/)（单个 exe，免安装）
3. 右键 zadig.exe →**以管理员身份运行**
4. 菜单 **Options → 勾选 List All Devices**（不勾看不到 iPhone）
5. 顶部下拉框选中 **iPhone**（可能显示为 Apple Mobile Device）
6. 箭头右侧目标驱动选 **libusbK**，点 **Replace Driver**，等约一分钟

> 副作用：换驱动后 iTunes/爱思助手暂时认不到手机，正常。
> 恢复方法：设备管理器 → 找到 iPhone → 右键更新驱动 → 选回苹果驱动。
> 随时可逆，手机本身不受任何影响。

**第 4 步：复查**

再次双击 `scripts\setup-windows.bat`（重复运行会跳过已装部分，直达自检），
这次预期**全绿**：

```
✓ pyusb 已安装
✓ libusb 后端可用
✓ 发现 1 个 Apple USB 设备
✓ 设备可访问 (权限/驱动正常)
✓ 00008xxx-xxxx  iPhone  QT配置: 未激活 (record 时会自动激活)
```

哪项是 ✗，它下面那行就是修复方法。全绿后跳到「3. 命令怎么敲」。

### 2.2 macOS

**第 1 步：跑一键脚本**

打开「终端」，进入项目目录执行：

```bash
cd ~/imirror        # 换成你的实际路径
bash scripts/setup-macos.sh
```

脚本自动完成：brew 装 libusb/uv/ffmpeg → 建虚拟环境 → 装依赖 →
跑 22 个离线测试 → 环境自检。

- 提示「未安装 Homebrew」→ 复制脚本给出的那行命令装好 brew，再重跑脚本
- 结尾自检显示「✗ 未发现 Apple USB 设备」→ 正常，还没插手机

**第 2 步：连接手机并信任**

1. 退出 QuickTime、爱思助手等可能占用 iPhone 的程序
2. 数据线连接 → 解锁 iPhone → 弹「信任此电脑？」→ 点**信任**

**第 3 步：复查**

```bash
.venv/bin/python -m imirror doctor
```

预期全绿（同 Windows 第 4 步的输出）。macOS 不需要 sudo、不需要换驱动。

> ⚠️ macOS 路线尚未真机验证过（macOS 自带投屏，本项目在 mac 上主要是开发调试用），
> 跑出问题优先反馈日志。

### 2.3 Linux

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh    # 没装 uv 先装
uv venv .venv
uv pip install --python .venv/bin/python -e ".[dev]"
.venv/bin/python -m pytest tests -q                # 预期 22 passed
```

USB 权限（一次性）：

```bash
sudo tee /etc/udev/rules.d/39-imirror.rules <<'EOF'
SUBSYSTEM=="usb", ATTR{idVendor}=="05ac", MODE="0666"
EOF
sudo udevadm control --reload-rules && sudo udevadm trigger
```

配置后重新插拔手机即可免 sudo；不想配就在每条命令前加 `sudo`。
播放验证工具：`sudo apt install ffmpeg`。连手机、信任，同 macOS 第 2 步。

## 3. 命令怎么敲（往下所有步骤通用）

不需要"激活虚拟环境"，直接用完整路径调用，进入**项目目录**后：

| 你的系统 | 命令前缀（下文用 `<py>` 代替） |
| --- | --- |
| Windows（PowerShell 或 CMD） | `.venv\Scripts\python.exe` |
| macOS / Linux（终端） | `.venv/bin/python` |

例如下文写 `<py> -m imirror doctor`，Windows 就敲：

```powershell
.venv\Scripts\python.exe -m imirror doctor
```

macOS/Linux 就敲：

```bash
.venv/bin/python -m imirror doctor
```

## 4. 分步联调

### 第 1 步：自检 + 看到设备

```
<py> -m imirror doctor
<py> -m imirror devices
```

**预期**：doctor 全绿；devices 输出一行设备：

```
00008110-000A1B2C3D4E5F  iPhone  vid:pid=05ac:12a8  QT配置: 未激活
```

**不对时**：doctor 每个 ✗ 下面都印了你系统对应的修复命令，照做后重跑。

### 第 2 步：激活投屏配置 ⚠️ 联调重点 1

> 这步可以跳过（record 会自动激活），但单独跑一次便于把问题定位在激活环节。

```
<py> -m imirror -v activate
```

**预期**：手机「断开又出现」一次（可能有提示音），几秒内打印：

```
xxx QT 配置已激活 (config #N)
```

再跑一次 `devices`，QT配置 应显示「已激活」。

**这一步验证代码里的第一个 TODO**：激活请求发出后设备重新枚举，
`activation.py` 轮询 10 次 × 0.5 秒等它回来。**不对时**：
- 报「无法激活」→ 重枚举比 5 秒慢。保留 `-v` 输出，并告知从手机断开到
  重新出现大概几秒（Linux 看 `dmesg -w`，Windows 听设备插拔提示音）
- Windows 激活后设备"消失" → 重枚举出的新配置可能也要在 Zadig 里再换一次驱动

### 第 3 步：录制 10 秒 ⚠️ 联调重点 2（最关键的一步）

录之前在手机上放段音乐/视频（方便后面验证音频），保持手机解锁：

```
<py> -m imirror -v record out.h264 out.wav --duration 10
```

**预期 `-v` 日志按这个顺序出现**（顺序就是协议握手）：

```
已 claim QuickTime 接口 #N (in:0x8x out:0x0x)
收到 PING, 回复 PING
发送 ASYN HPD1 x2
回复 CWPA clockRef=xxxx
发送 ASYN HPA1
发送首个 NEED, clockRef=xxxx
音频格式: AudioStreamBasicDescription{...48000...}
写入参数集: FormatDescriptor{video 886x1920, codec:avc1, sps:xxB, pps:xxB}   ← 最重要的一行
  [    5s] 视频 xxx 帧 (~30.0fps, x.xMB)  音频 xxx 帧 (xxxKB)
  [   10s] ...
请求设备停止推流...
会话已关闭 (视频帧:N 音频帧:M)
```

**核对点**：
1. `写入参数集` 这行必须出现，且 sps/pps 字节数 > 0
   —— 这验证 `formatdescriptor.py` 的 SPS/PPS 提取位置（代码里标了 TODO 的地方）
2. 统计行的视频帧数持续增长（说明 FEED→NEED 流控正常）
3. 结束时「会话已关闭」前**没有** RELS 超时告警
   —— 这验证按 Go 版修正过的 `close_session` 关闭时序

**不对时**（保留完整日志，见第 6 节的收集命令）：
- 卡在 claim 接口 / 报 `Resource busy` → Windows: 确认 Apple Mobile Device
  Service 已停用；macOS: 退出 QuickTime 等；Linux: `sudo systemctl stop usbmuxd`
- 收到 PING 后没有下文 → 可能是分帧或首包问题，发日志
- 没有「写入参数集」→ **这就是 TODO 命中的情况**，日志是修复
  `formatdescriptor.py` 需要的第一手数据，务必保留
- 帧数不增长/几秒后停 → FEED/NEED 流控问题，同样发日志

### 第 4 步：验证录制产物

```
ffplay -f h264 out.h264        # 应看到手机屏幕画面
ffprobe -f h264 out.h264       # 应显示正确分辨率
ffplay out.wav                 # 应听到手机声音
```

（`ffplay` 命令不存在：Windows 重开一个终端窗口再试，还不行用 VLC
直接打开这两个文件；macOS `brew install ffmpeg`；Ubuntu `sudo apt install ffmpeg`。）

**核对点**：画面不花屏、分辨率与手机匹配、音频时长 ≈ 10 秒。

- 画面花屏/绿屏但能播 → SPS/PPS 内容可能不对，保留 out.h264 和日志
- wav 静音 → 确认录制时手机在放声音；仍静音则保留 wav 和日志

### 第 5 步：实时预览（可选）

```
# Windows
uv pip install --python .venv\Scripts\python.exe -e ".[gui]"
# macOS/Linux
uv pip install --python .venv/bin/python -e ".[gui]"

<py> -m imirror gui            # 按 q 退出
```

**预期**：弹出窗口显示手机画面，延迟 < 1 秒。卡顿属于已知待优化项。

### 第 6 步：恢复手机正常模式

正常退出后设备自动释放；电脑不识别手机时重新插拔数据线即可。
Windows 想恢复 iTunes 同步：设备管理器把 iPhone 驱动换回苹果的。

## 5. 联调验收清单

全部打勾即可宣布真机联调完成，把结果（含操作系统与 iOS 版本）同步到
AGENTS.md 的「当前状态」：

- [ ] `doctor` 全部 ✓
- [ ] `devices` 能列出设备
- [ ] `activate` 激活成功（重枚举 ≤ 5 秒）
- [ ] `record` 完整跑通 10 秒无报错
- [ ] 日志出现「写入参数集」且 sps/pps > 0 字节（formatdescriptor.py TODO 销案）
- [ ] `ffplay` 能正常播放 out.h264，画面正确
- [ ] out.wav 有声音
- [ ] 结束时无 RELS 超时告警（close_session 时序验证）
- [ ] 连续录 3 分钟以上不断流（SKEW 应答验证）

## 6. 出问题时收集什么

把以下产物打包反馈，能极大加快远程定位：

```
# 1. 环境自检结果
<py> -m imirror doctor > doctor.txt 2>&1

# 2. 完整 DEBUG 日志
<py> -m imirror -v record out.h264 out.wav --duration 10 > record.log 2>&1

# 3. 系统侧 USB 信息
#    Windows: 设备管理器截图(iPhone 所在节点 + 当前驱动名)
#    macOS:   system_profiler SPUSBDataType > usb.txt
#    Linux:   lsusb -d 05ac: -v > usb.txt 2>/dev/null; dmesg | tail -100 > dmesg.txt
```

外加一句话描述：卡在第几步、现象是什么、手机型号和 iOS 版本。
