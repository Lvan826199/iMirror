# 真机联调手册

从零开始，Windows / macOS / Linux 三个系统的完整步骤。
每一步都写了**要执行的命令、预期结果、不对时看什么**，不需要懂 Python。

## 0. 你需要什么

- 一台 iPhone 或 iPad（不需要越狱，系统版本不限，越新越有验证价值）
- 一根**支持数据传输**的数据线（纯充电线不行，这是最常见的坑）
- Windows / macOS / Linux 电脑任意一台（要有装软件的权限）

## 1. 下载项目

**方式 A：用 git（装过 git 的话）**

```bash
git clone https://gitee.com/xiaozai-van-liu/imirror.git
```

**方式 B：下载 ZIP（没装 git）**

浏览器打开 https://gitee.com/xiaozai-van-liu/imirror ，
点右上角橙色「克隆/下载」→「下载 ZIP」，解压。

> Windows 建议解压到**不带中文和空格**的路径，如 `D:\imirror`；
> macOS 随意，如 `~/imirror`。

## 2. 安装环境（按你的系统看对应小节，其余跳过）

### 2.1 Windows

**第 1 步：双击运行 `scripts\setup-windows.bat`**

脚本自动完成：装 uv（自带 Python，无需单独装）→ 建虚拟环境 → 装依赖
（含 libusb，无需手动放 dll）→ 跑 32 个离线测试 → 装 ffmpeg → 环境自检。
全程几分钟，不用操作。

预期窗口结尾（此刻还没插手机/没换驱动，最后一个 ✗ 是正常的）：

```
✓ pyusb 已安装
✓ libusb 后端可用
✗ 未发现 Apple USB 设备 (vid=05ac)     ← 正常
```

常见问题：
- 双击弹出「Windows 已保护你的电脑」→ 点「更多信息」→「仍要运行」
- winget 装 ffmpeg 失败 → 不影响采集，只影响最后播放验证，可后面用 VLC 代替
- 显示「[失败] 安装中断」→ 把窗口内容完整复制反馈

**第 2 步：连接手机并信任（⚠️ 必须在换驱动之前做）**

1. 数据线连接 iPhone 和电脑
2. 解锁 iPhone → 弹出「要信任此电脑吗？」→ 点**信任**，输一次锁屏密码
3. 没弹？拔掉重插一次，保持手机解锁

> 顺序很重要：信任弹窗依赖苹果原始驱动，要趁驱动还没换的时候完成。

**第 3 步：换驱动（唯一需要手动点的环节，一次性）**

> 如果电脑同时插着多台苹果设备（iPad、另一台 iPhone），先拔掉不测的，避免换错。
> 驱动映射是一台设备一份：只换 iPhone 不影响 iPad 与 iTunes 的连接；
> 以后想采集 iPad，就对它的条目（如 USB ID `05AC 12AB`）重复同样操作，
> 多台同时接时录制用 `--udid 序列号` 指定设备（序列号用 `devices` 命令查）。

1. `Win + R` → 输入 `services.msc` 回车 → 找到 **Apple Mobile Device Service**
   → 右键**停止** → 双击它，启动类型改为**禁用**
   （列表里没有这个服务？说明没装过 iTunes，直接进入下一小步）
2. 右键仓库自带的 `scripts\zadig-2.9.exe` →**以管理员身份运行**
   （或从 https://zadig.akeo.ie/ 下载最新版）
3. 菜单 **Options**，做两件事：
   - ✅ 勾选 **List All Devices**
   - ❌ **取消勾选 Ignore Hubs or Composite Parents**（⚠️ 关键！iPhone 是
     复合设备，这项默认勾着会把需要换驱动的"复合父设备"藏起来——
     只换某个子接口是不生效的）
4. 顶部下拉框**按 USB ID 列选择**：找 **`05AC 12A8`** 的条目
   （名字可能显示 Apple iPhone / Apple Mobile Device USB Composite Device /
   或只显示 Apple，都正常；出现多个 05AC 条目时，选标注
   **Composite Parent** 的那个）
5. 设置目标驱动并执行替换。Zadig 中间有**两个驱动框**：
   - **左边框** = 设备当前的驱动（显示什么都不用管）
   - **右边框** = 要换成的目标驱动，用旁边的**上下小箭头**切换，
     候选有四个，**选 `libusb-win32 (v1.x.x.x)`**：

   | 候选驱动 | 选不选 | 说明 |
   | --- | --- | --- |
   | **libusb-win32** | ✅ **选这个** | 完整内核驱动，**支持切换 USB 配置**——iPhone 的投屏接口在非默认配置上，必须靠它切过去。原版 C++ 项目用的也是它 |
   | libusbK | ❌ | 能枚举设备、能激活，但**不支持切配置**，一到录制就报 `set_configuration ... not supported`（我们踩过这个坑） |
   | WinUSB | ❌ | 同 libusbK 的限制，且复合设备支持更差 |
   | USB Serial (CDC) | ❌ | 串口驱动，与本项目无关 |

   切到 libusb-win32 后点中间的大按钮 **Replace Driver**
   （按钮显示 Install Driver / Reinstall Driver 也一样点），等约一分钟。

> **为什么是 libusb-win32 而不是 libusbK？** iPhone 有多个 USB 配置，投屏接口
> 位于非默认配置（实测在配置 #5，默认活动的是 #1）。Windows 上只有 libusb-win32
> 这种完整驱动能切换活动配置；libusbK/WinUSB 会在切配置时返回 NOT_SUPPORTED。
> 若你之前换成了 libusbK，重跑 Zadig 换成 libusb-win32 即可。

**怎么确认换好了（任选其一）**：
- Zadig 里重新选中该设备，**左边** Driver 框显示的是当前驱动——
  写 `libusb-win32 (v...)` 即成功；写 `usbccgp` 或 `Apple Mobile Device` 即没换到位
- 设备管理器（`Win+X` → 设备管理器）出现新分类 **libusb-win32 devices**（或
  **libusb (WinUSB)** 之外的 libusb-win32 节点），iPhone 在它下面即成功
- 跑 doctor 能识别设备，且随后 `record`/`gui` 不再报 `set_configuration not supported`
  （最终标准）

> 副作用：换驱动后 iTunes/爱思助手暂时认不到手机，正常。
> 恢复方法：设备管理器 → 找到 iPhone → 右键更新驱动 → 选回苹果驱动。
> 随时可逆，手机本身不受任何影响。

**第 4 步：复查**

再次双击 `scripts\setup-windows.bat`（重复运行会跳过已装部分，直达自检），
这次预期**全绿**：

```
✓ pyusb 已安装
✓ libusb 后端可用
✓ 发现 1 个 Apple USB 设备
✓ 设备可访问 (权限/驱动正常)
✓ 00008xxx-xxxx  iPhone  QT配置: 未激活 (record 时会自动激活)
```

哪项是 ✗，它下面那行就是修复方法。全绿后跳到「3. 命令怎么敲」。

### 2.3 Linux

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh    # 没装 uv 先装
uv venv .venv
uv pip install --python .venv/bin/python -e ".[dev]"
.venv/bin/python -m pytest tests -q                # 预期 32 passed
```

USB 权限（一次性）：

```bash
sudo tee /etc/udev/rules.d/39-imirror.rules <<'EOF'
SUBSYSTEM=="usb", ATTR{idVendor}=="05ac", MODE="0666"
EOF
sudo udevadm control --reload-rules && sudo udevadm trigger
```

配置后重新插拔手机即可免 sudo；不想配就在每条命令前加 `sudo`。
播放验证工具：`sudo apt install ffmpeg`。连手机、信任，同 macOS 第 2 步。

## 3. 命令怎么敲（往下所有步骤通用）

不需要"激活虚拟环境"，直接用完整路径调用，进入**项目目录**后：

| 你的系统 | 命令前缀（下文用 `<py>` 代替） |
| --- | --- |
| Windows（PowerShell 或 CMD） | `.venv\Scripts\python.exe` |
| macOS / Linux（终端） | `.venv/bin/python` |

例如下文写 `<py> -m imirror doctor`，Windows 就敲：

```powershell
.venv\Scripts\python.exe -m imirror doctor
```

macOS/Linux 就敲：

```bash
.venv/bin/python -m imirror doctor
```

## 4. 多设备时指定设备（--udid）

同时接多台苹果设备时，`record`/`gui`/`activate` 都要加 `--udid 序列号`
指定操作哪台（序列号从 `doctor` 或 `devices` 的输出里复制）。
新版 CLI 在识别到多台 iOS 设备且未传 `--udid` 时会直接停止并列出可选序列号，
避免默认选第一台导致激活或录制跑到错误设备。

以本项目联调用的三台设备为例（Windows 命令，可直接复制改序列号）：

| 设备 | 序列号（udid） |
| --- | --- |
| iPhone ① | `00008110000275943EEB801E` |
| iPhone ② | `00008101001859DE1E38001E` |
| iPad | `49687f67a4c70fbd027e19b4a5e40218acdc06e4` |

```powershell
# 录 iPhone ① 10 秒(验证链路)
.venv\Scripts\python.exe -m imirror -v record out.h264 out.wav --duration 10 --udid 00008110000275943EEB801E

# 实时投屏 iPhone ① (窗口按 q 退出; 首次先装 gui 依赖, 见第 5 步)
.venv\Scripts\python.exe -m imirror gui --udid 00008110000275943EEB801E

# 录 iPad
.venv\Scripts\python.exe -m imirror -v record ipad.h264 ipad.wav --duration 10 --udid 49687f67a4c70fbd027e19b4a5e40218acdc06e4
```

> 说明：`record`（存成文件）和 `gui`（实时投屏窗口）走的是同一条数据链路，
> 只是出口不同。联调先跑 record 是因为它不依赖解码/显示，问题好定位；
> record 通了 gui 就是换个命令的事。

## 4.1 换机验证顺序（Windows → Ubuntu）

macOS 已经证明设备、系统原生 QuickTime 链路和基础协议方向可行；换电脑后不要先开
GUI，先用 `record --duration 10` 把 USB/协议链路压到最小变量。

**Windows 首测顺序**：

```powershell
.venv\Scripts\python.exe -m imirror doctor
.venv\Scripts\python.exe -m imirror devices --json
.venv\Scripts\python.exe -m imirror -v record out.h264 out.wav --duration 10 --udid 设备序列号
```

判读重点：
- `doctor` 读不到序列号或全部节点不可访问：Zadig 没换到复合父设备，或激活后的新形态
  还没换 libusb-win32。
- `devices --json` 里 `qt_available=true` 但 `qt_enabled=false`：能看到 QT 描述符，
  但当前活动配置还没切过去。Windows 新版会优先重发 Apple QT enable，让设备按原始
  激活路径重新枚举并进入会话起点；若仍激活失败，再按 Zadig 二次换驱动处理。
- 日志出现 `已 claim QuickTime 接口` 后没有 `收到 PING`：进入 Windows 最关键验证点。
  继续看后面是否出现 `唤醒敲门` 和 `已主动发送 PING`；若只看到 `探测写 USB 失败`
  也属于端点仍未唤醒的有效诊断，不会再卡死。若敲门后仍 0 帧，把完整
  `record.log`、设备管理器里 iPhone 当前驱动名、Zadig 当前左右驱动框截图一起保存。
- 日志已经出现 `收到 PING, 回复 PING` 后又停住：这说明 bulk IN/OUT 曾经通。新版代码
  不再继续敲门打断会话；请更新后复测, 重点看 PING 后是否继续出现 `SYNC OG/CWPA/CVRP`。
- 上一轮失败后 `devices` 显示 QT 已激活, 但下一次 `record` 直接打开后完全没有 PING：
  这是 Windows 半激活残留态。新版 `record/gui/activate` 会在 Windows 上强制重发 QT enable
  重新武装会话；若当前仍在普通配置 #1 但已经暴露 QT 配置 #5，也会优先走 vendor enable，
  不再先用 `set_configuration(5)` 制造“能 claim 但没有首个 PING”的假激活态。
- 一旦 10 秒 `record` 出现 `写入参数集` 且视频帧增长，再跑 `gui`；GUI 只是出口验证。

**Ubuntu 首测顺序**：

```bash
sudo apt update
sudo apt install -y libusb-1.0-0 ffmpeg
curl -LsSf https://astral.sh/uv/install.sh | sh
uv venv .venv
uv pip install --python .venv/bin/python -e ".[dev,gui]"
.venv/bin/python -m pytest tests -q
.venv/bin/python -m imirror doctor
.venv/bin/python -m imirror -v record out.h264 out.wav --duration 10 --udid 设备序列号
```

Ubuntu 比 Windows 简单，通常只卡权限或占用：
- `Access denied`：先用 `sudo .venv/bin/python -m imirror doctor` 验证；能通就加 udev 规则。
- `Resource busy`：临时 `sudo systemctl stop usbmuxd` 后重试。
- claim 成功但无 PING：保留 `record.log`、`lsusb -d 05ac: -v` 和
  `dmesg | tail -100`，这个结果比 Windows 更有定位价值，因为 Linux 没有 Zadig/驱动实例变量。

## 5. 分步联调

下面的 `record`/`gui` 是跨平台 raw USB QuickTime 协议路径；macOS 另有实验性的
系统原生路径，见本节末尾「macOS 原生后端」。

### 第 1 步：自检 + 看到设备

```
<py> -m imirror doctor
<py> -m imirror devices
```

**预期**：doctor 全绿；devices 输出一行设备：

```
00008110-000A1B2C3D4E5F  iPhone  vid:pid=05ac:12a8  QT配置: 未激活
```

**不对时**：doctor 每个 ✗ 下面都印了你系统对应的修复命令，照做后重跑。

### 第 2 步：激活投屏配置 ⚠️ 联调重点 1

> 这步可以跳过（record 会自动激活），但单独跑一次便于把问题定位在激活环节。

```
<py> -m imirror -v activate
```

**预期**：手机「断开又出现」一次（可能有提示音），几秒内打印：

```
xxx QT 配置已激活 (config #N)
```

再跑一次 `devices`，QT配置 应显示「已激活」。

macOS 上如果看到「可用但未激活 (active #5, QT #6)」，表示设备仍能看到
Valeria/QT 描述符，但当前活动配置已经回到普通 usbmux 配置；这是可恢复状态，
再次运行 `activate` 或 `record` 会直接切回 QT 配置。若怀疑系统/设备卡在半激活
状态，可先跑 `<py> -m imirror reset`，它会对 iPhone 发 USB reset 并等待重新枚举。

**这一步验证代码里的第一个 TODO**：激活请求发出后设备重新枚举，
`activation.py` 轮询 30 次 × 0.5 秒等它回来。**不对时**：
- **Windows 报 `RuntimeError: 无法为 xxx 激活 QuickTime 配置`（最常见）**：
  激活后 iPhone 以"新形态"重新枚举，Windows 给它派了默认驱动，libusb 看不到。
  **自救**：保持手机连接 →（管理员）重开 Zadig，确认 Options 两个设置
  （勾 List All Devices、取消勾 Ignore Hubs or Composite Parents）→
  下拉框找**新出现的** iPhone 条目，再对它换一次 **libusb-win32** → 跑 doctor，
  这次 QT配置 应显示「已激活」→ 继续录制/投屏。这次换完以后就不用再换了
- Linux/macOS 报「无法激活」→ 重枚举比 15 秒慢，属异常。保留 `-v` 输出，
  并告知从手机断开到重新出现大概几秒（Linux 看 `dmesg -w`）

### 第 3 步：录制 10 秒 ⚠️ 联调重点 2（最关键的一步）

录之前在手机上放段音乐/视频（方便后面验证音频），保持手机解锁：

```
<py> -m imirror -v record out.h264 out.wav --duration 10
```

**预期 `-v` 日志按这个顺序出现**（顺序就是协议握手）：

```
已 claim QuickTime 接口 #N (in:0x8x out:0x0x)
收到 PING, 回复 PING
发送 ASYN HPD1 x2
回复 CWPA clockRef=xxxx
发送 ASYN HPA1
发送首个 NEED, clockRef=xxxx
音频格式: AudioStreamBasicDescription{...48000...}
写入参数集: FormatDescriptor{video 886x1920, codec:avc1, sps:xxB, pps:xxB}   ← 最重要的一行
  [    5s] 视频 xxx 帧 (~30.0fps, x.xMB)  音频 xxx 帧 (xxxKB)
  [   10s] ...
请求设备停止推流...
会话已关闭 (视频帧:N 音频帧:M)
```

**核对点**：
1. `写入参数集` 这行必须出现，且 sps/pps 字节数 > 0
   —— 这验证 `formatdescriptor.py` 的 SPS/PPS 提取位置（代码里标了 TODO 的地方）
2. 统计行的视频帧数持续增长（说明 FEED→NEED 流控正常）
3. 结束时「会话已关闭」前**没有** RELS 超时告警
   —— 这验证按 Go 版修正过的 `close_session` 关闭时序

**不对时**（保留完整日志，见第 6 节的收集命令）：
- 卡在 claim 接口 / 报 `Resource busy` → Windows: 确认 Apple Mobile Device
  Service 已停用；macOS: 退出 QuickTime 等；Linux: `sudo systemctl stop usbmuxd`
- macOS claim 成功但 10 秒内没有 `收到 PING`，并且日志只有 `读超时` / bulk 写
  `Operation timed out` → 先确认手机保持解锁且已信任；换成 Mac 直连或高速 USB 口，
  避免接在键盘/低速 Hub 上（`system_profiler SPUSBDataType` 里 iPhone 不应只显示
  `Up to 12 Mb/s`）；再跑 `<py> -m imirror reset` 清掉残留 QT 描述符/半激活状态后
  重试。修复后的 `record` 会在 0 视频帧时返回失败，仍复现则保留
  `record -v` 日志和 `system_profiler` 输出。若系统原生路线可接受，macOS 还需要
  CoreMediaIO 打开 `AllowScreenCaptureDevices` 并给运行进程 Screen Recording 权限；
  无权限时系统日志会出现 `kTCCServiceScreenCapture` 拦截。
- 收到 PING 后没有下文 → 可能是分帧或首包问题，发日志
- 没有「写入参数集」→ **这就是 TODO 命中的情况**，日志是修复
  `formatdescriptor.py` 需要的第一手数据，务必保留
- 帧数不增长/几秒后停 → FEED/NEED 流控问题，同样发日志

### 第 4 步：验证录制产物

```
ffplay -f h264 out.h264        # 应看到手机屏幕画面
ffprobe -f h264 out.h264       # 应显示正确分辨率
ffplay out.wav                 # 应听到手机声音
```

（`ffplay` 命令不存在：Windows 重开一个终端窗口再试，还不行用 VLC
直接打开这两个文件；macOS `brew install ffmpeg`；Ubuntu `sudo apt install ffmpeg`。）

**核对点**：画面不花屏、分辨率与手机匹配、音频时长 ≈ 10 秒。

- 画面花屏/绿屏但能播 → SPS/PPS 内容可能不对，保留 out.h264 和日志
- wav 静音 → 确认录制时手机在放声音；仍静音则保留 wav 和日志

### 第 5 步：实时预览（可选）

用一键脚本装的环境**已自带 GUI 依赖**，直接跑；手动装环境的先补装：

```
# 仅手动装环境时需要 —— Windows:
uv pip install --python .venv\Scripts\python.exe -e ".[gui]"
#                        macOS/Linux:
uv pip install --python .venv/bin/python -e ".[gui]"

<py> -m imirror gui            # 按 q 退出
```

> ⚠️ 不要运行 `uv sync`——它会把环境"精确同步"成只剩基础依赖，
> 把 dev/windows/gui 附加项全部卸载。误跑了的话重新双击一键脚本即可恢复。

**预期**：弹出窗口显示手机画面，延迟 < 1 秒。卡顿属于已知待优化项。

### macOS 原生后端（实验性）

macOS 上如果 raw USB `record` 已经能 claim 接口但一直没有 `收到 PING`，可以用系统
CoreMediaIO/AVFoundation 后端交叉验证。它会先打开系统
`AllowScreenCaptureDevices` 开关，让 iPhone 作为 `.muxed` 屏幕源出现：

```
<py> -m imirror macos-devices
<py> -m imirror macos-record native.mov --duration 10
<py> -m imirror macos-gui
```

首次运行会用 `swiftc` 把 helper 编译并 ad-hoc 签名到 `~/Library/Caches/imirror/`，
后续复用同一可执行文件，减少 `swift-frontend` 解释执行带来的 TCC 身份干扰。

预期 `macos-devices` 能看到类似：

```
00008110-000275943EEB801E  唐三的iPhone  model=iOS Device  type=AVCaptureDeviceTypeExternal
```

如果 `macos-record` 报 `Screen Recording permission denied`，到系统设置的
「隐私与安全性 → 屏幕录制」里给启动 iMirror 的 App/终端授权，授权后重启该 App/终端。
如果报 `recording failed before start` 或 `valeria connection not seen`，表示系统
`iOSScreenCaptureAssistant` 也没能让手机进入 Valeria 出流态；保持手机解锁、重插数据线、
退出 QuickTime/录屏占用程序后再试，并保留 `log show` 里的 iOSScreenCaptureAssistant 日志。
如果 QuickTime Player 自己在选择 iPhone 后也弹「这项操作无法完成」，并且日志出现
`StartStream throwing valeria connection not seen` / `AVFoundationErrorDomain -11800`
或 `-11806`，先把项目路径排除掉：跑 `<py> -m imirror reset`，重启 Mac 和 iPhone，
换 Mac 直连口/数据线，再用 QuickTime 复验。QuickTime 仍失败时，当前 Mac/iPhone 的
系统 Valeria 链路没有建立，iMirror 暂时无法绕过这层系统失败。
如果同一台 Mac 上多台设备都能投屏，但某一台设备突然失败，优先按“设备/macOS
Valeria 临时状态卡死”处理：保存失败设备的 `log show` 关键行后，依次尝试
`<py> -m imirror reset`、重插数据线、重启 iPhone、重启 Mac。重启后恢复时，不要再把
问题归因到线材、基础权限、Python 报文序列化或 iOS 大版本兼容性。

### 第 6 步：恢复手机正常模式

正常退出后设备自动释放；电脑不识别手机时重新插拔数据线即可。macOS/Linux 也可先试：

```
<py> -m imirror reset
```

该命令用于恢复半激活状态，成功后 `devices` 应回到「未激活」或「可用但未激活」。
Windows 想恢复 iTunes 同步：设备管理器把 iPhone 驱动换回苹果的。
