# 已知问题与归因

本文件记录真机联调阶段已经踩过的坑，目的是避免重复误判。

当前交付状态：

- macOS 原生 CoreMediaIO/AVFoundation 投屏已跑通，是当前唯一验收主线。
- Windows 投屏兼容已暂停，不再作为当前交付目标。
- MSCA 集成已暂停，iMirror 不负责 WDA 控屏。

## A 类：pyusb 异常捕获不完整

现象：`doctor`、`devices`、`record` 或 `gui` 直接抛 traceback，而不是给出明确提示。

根因：pyusb 底层不只抛 `usb.core.USBError`。读字符串描述符可能抛 `ValueError`，
后端不支持的操作可能抛 `NotImplementedError`。

防复发：

- 任何 pyusb 调用点捕获异常时，一律写全：

```python
except (usb.core.USBError, NotImplementedError, ValueError):
```

- 不要只捕获 `USBError`。
- 关闭会话时不能假设握手已经完成，也不能假设 bulk 端点仍可写。

## B 类：USB 字符串描述符包含不可见字符

现象：`--udid` 明明来自 `devices` 输出，但仍匹配不到设备。

根因：USB 字符串描述符可能包含 NUL 填充或格式差异。

防复发：

- 从 USB 描述符读出的字符串统一清理 NUL 和空白。
- 面向用户的 UDID 匹配要兼容带横杠和不带横杠两种格式。
- 多设备场景不要默认选第一台，必须提示用户显式传 `--udid`。

## C 类：macOS Valeria 临时状态卡死

现象：

- `macos-devices` 能看到设备，但 `macos-record` 报 `recording failed before start`。
- 系统日志出现 `StartStream throwing valeria connection not seen`。
- QuickTime Player 选择 iPhone 也可能提示“这项操作无法完成”。

根因：macOS 的 `iOSScreenCaptureAssistant` 没能让设备进入 Valeria 出流状态。
这通常是设备/macOS 原生链路状态问题，不是 Python 报文序列化或端点选择问题。

处理顺序：

1. 保持设备解锁并信任电脑。
2. 退出 QuickTime、会议软件、录屏工具等占用程序。
3. 运行 `python -m imirror reset`。
4. 重插数据线。
5. 用 QuickTime Player 交叉验证。
6. 重启 iPhone。
7. 重启 Mac。

防复发：

- 若 QuickTime Player 也失败，不要继续改 iMirror 协议代码。
- 若同一台 Mac 上其他设备正常，优先判断为单台设备临时状态卡死。

## D 类：QuickTime 配置状态判定混淆

现象：`devices` 显示能看到 QT/Valeria 配置描述符，但实际 record 没有出流。

根因：“描述符存在”不等于“当前活动配置已切到 QT”。设备可能保留 QT 描述符，
但 active configuration 已经回到普通 usbmux 配置。

防复发：

- 日志和文档必须区分 `qt_available` 和 `qt_enabled`。
- 需要判断当前 active configuration，而不是只看是否能读到 QT 描述符。
- 半激活状态优先用 `imirror reset` 清理。

## E 类：Windows 历史问题（已暂停）

Windows raw USB、chotgpt tools、爱思/Rayren、MSCA/WDA 共存路线均已暂停。
以下结论只作为历史记录：

- Windows USB 栈与 macOS/Linux 差异很大。
- 普通 WinUSB/libusbK 无法可靠支持 iPhone QuickTime 所需的非默认配置切换。
- libusb0/过滤驱动方案会引入设备重枚举、Lockdown/usbmux 抢占和稳定性问题。
- Rayren/爱思类有线视频与 WDA 控制在本机验证中存在 USB/Lockdown 资源冲突。

防复发：

- 不再给普通用户写 Windows 驱动配置教程。
- 不再把 Windows 作为当前兼容目标。
- 后续如果重启 Windows 方向，必须重新立项，独立评估，不沿用旧实验流程作为产品文档。

## 通用规约

1. 改 CLI 参数、依赖、协议逻辑时，必须同步 README、真机手册、协议速查和 AGENTS。
2. 修 bug 时必须在本文件补充现象、根因和 commit。
3. 真机问题先做交叉验证：QuickTime Player、系统日志、换线、换设备、重启设备。
