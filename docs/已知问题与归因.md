# 已知问题与归因（防复发清单）

真机联调阶段暴露的 bug 全部记录在此，按**根因**归类。改代码/加功能前先扫一遍
对应类别的"防复发规约"，避免重蹈覆辙。每条附修复 commit 便于回溯。

---

## A 类：pyusb 异常类型捕获不全 ⭐ 最高频

**现象**：doctor / gui / 打开设备时直接抛 Traceback 崩溃，而非给出友好提示。

**已发生的具体 bug**：
| 现象 | 抛出的异常 | 修复 commit |
| --- | --- | --- |
| Windows 未换驱动，doctor 读序列号崩溃 | `ValueError: has no langid` | eb5d5de |
| gui 缺 av/cv2 依赖时崩栈 | `ImportError`（且 import 在函数体内，except 位置错） | e8b584a |
| Windows 切 USB 配置崩溃 | `NotImplementedError: Operation not supported` | 6a56b3a |
| macOS claim 成功但 bulk 无数据，record 停止时崩栈 | `USBTimeoutError: Operation timed out` | 本次提交 |
| macOS 自动激活后立即打开设备，claim 接口崩栈 | `USBError: [Errno 2] Entity not found` | 本次提交 |
| macOS USB reset 触发重新枚举时报错但实则成功 | `USBError: [Errno 2] Entity not found` | 本次提交 |

**根因**：pyusb 底层（libusb1 backend）**不只抛 `usb.core.USBError`**。读字符串描述符
失败抛 `ValueError`，后端不支持的操作抛 `NotImplementedError`，还有 `NotImplementedError`
被包在 `_check()` 里。只捕获 `USBError` 必然漏网。

**防复发规约**：
- 任何 pyusb 调用点（`get_string` / `set_configuration` / `ctrl_transfer` /
  `read` / `write` / `get_active_configuration`），捕获异常一律写全三元组：
  `except (usb.core.USBError, NotImplementedError, ValueError)`。
- 延迟 import 的可选依赖（av/cv2），try 要**把使用点也包进去**，不能只包 import 行。
- 关闭会话不能假设握手已经完成或 bulk 端点仍可写；停止报文发送失败时应记录告警、
  停止 consumer 并输出统计，避免丢失前面的真实 USB 日志。
- 激活后能读到 QT 配置描述符不代表接口已可 claim；自动 `record`/`gui` 路径要留
  settle 时间，claim 失败时短重试并报明确错误。
- `record` 结束时若视频帧数为 0，应返回失败，避免把空 `.h264` 和仅 WAV 头误报为可播放产物。

## G 类：macOS Valeria 临时状态卡死

**现象**：macOS 上 `doctor` 全绿，`activate` 成功，`record -v` 能切到 QT 配置并
claim QuickTime 接口，但 10 秒内没有 PING/FEED/EAT，bulk 读持续超时，停止时 bulk
写 HPA0/HPD0 也超时，产物为 0 字节 `.h264` 与仅 WAV 头。

**本次现场信息（2026-07-10）**：
- macOS 15.7.7，Python 3.10.11，`libusb-package 1.0.30.0`
- iPhone `05ac:12a8`，iOS 18.3，ProductType `iPhone14,4`，QT 配置 #6，
  接口 #2，端点 `in:0x86 out:0x05`
- 初次 `system_profiler SPUSBDataType` 显示 iPhone 挂在 USB Hub 下且只有 `Up to 12 Mb/s`；
  后续换到 `Up to 480 Mb/s` 高速连接后，bulk 仍然双向超时
- Windows 参考里的“唤醒敲门”（vendor `0x40/0x40/0x6400/0x6400` + 主动 PING）
  在本机上控制请求成功，但 PING bulk 写仍超时
- `iOSScreenCaptureAssistant` 是系统 LaunchDaemon；手动启动后会连 usbmuxd/lockdown，
  但 raw USB `record` 仍无 PING。助手符号显示现代 macOS 路线会走
  `APValeriaHelperActivationStart` / `APValeriaHelperReceiverStart` /
  `FigNeroStartStream`，不是单纯 claim bulk 后等待。
- CoreMediaIO 需要先设置 `kCMIOHardwarePropertyAllowScreenCaptureDevices` (`'yes '`)
  才会把 iPhone 暴露为 AVFoundation `.muxed` 设备；本机曾枚举到
  `唐三的iPhone / 00008110-000275943EEB801E`。
- 已新增实验性 macOS 原生命令：`macos-devices` 可稳定列出 `.muxed` 设备；
  `macos-record` 已走到 AVFoundation/iOSScreenCaptureAssistant 的 `StartStream`，
  但本机返回 `AVFoundationErrorDomain Code=-11806`，系统日志为
  `StartStream throwing valeria connection not seen`，未生成有效 `.mov`。
- 2026-07-10 复测：`imirror reset` 可把设备从“QT #6 描述符残留、active #5”的
  状态恢复到干净普通枚举（`bNumConfigurations=5`、`qt_config=-1`）；从干净状态跑
  `macos-record` 仍失败，系统日志显示 Valeria 接口已注册、discovery 已启动并
  `Valeria found an Octavia USB device`，随后 `StartStream throwing valeria connection not seen`。
- `macos-*` 命令已改为优先编译、ad-hoc 签名并复用 `~/Library/Caches/imirror/`
  下的 Swift helper，避免每次以临时脚本和 `swift-frontend` 作为 TCC accessing
  process；编译失败才回退到 `swift` 解释执行。
- 签名 helper 复测后, TCC attribution 已变为缓存 helper 的稳定代码身份, 但
  `macos-record` 仍在同一位置失败；因此当前主阻塞不是 `swift-frontend` 或未签名
  临时脚本, 仍是设备/系统未完成 Valeria connection。
- 2026-07-10 QuickTime Player 复测：用户重置 iPhone「位置与隐私」并重新信任后，
  QuickTime 新建影片录制选择 `唐三的iPhone` 仍弹「这项操作无法完成」。系统日志显示
  QuickTime 抛 `AVFoundationErrorDomain -11800` / underlying `OSStatus -308`；
  `iOSScreenCaptureAssistant` 先记录 `Valeria found an Octavia USB device`，随后
  `StartStream throwing valeria connection not seen`。同一轮里 `imirror reset` 后
  再跑 `macos-record` 仍为 `AVFoundationErrorDomain -11806 Recording Stopped`，
  raw USB `record` 仍能激活 #6 并 claim 接口 #2，但 8 秒内无 PING/FEED，bulk
  写 HPA0/HPD0 超时。
- 2026-07-10 交叉验证：同一台 Mac、同一套项目环境、同一类 USB 连接下，iOS
  14.4.2、15.1、16.7.8、17.7.4 的 iPad、iOS 26.2 均可 QuickTime 投屏成功；
  `iOSScreenCaptureAssistant` 日志出现 sample buffer 序列化。此前失败的 iOS
  18.3 设备在重启后也可 QuickTime 投屏，`devices --json` 显示已处于 QT config #6，
  `macos-devices` 能枚举到 `.muxed` 设备。
- 2026-07-10 iOS 18.3 设备重启恢复后，`imirror macos-gui --udid
  00008110-000275943EEB801E` 可成功启动原生预览；进程为缓存 helper
  `macos_native_cfd0c46c331b9680` + `iOSScreenCaptureAssistant`，系统日志出现
  `Valeria found an Octavia USB device` 和 sample buffer 序列化，不再出现
  `StartStream throwing valeria connection not seen`。

**当前判断**：协议层和端点选择未见异常，阻塞点在 USB bulk 传输建立前。交叉验证
已经把 Mac、线材、QuickTime 权限、CoreMediaIO 开关、iOS 大版本兼容性和项目基础环境
基本排除；失败更像是某次连接后设备/macOS 的 Valeria 出流状态卡死。现代 macOS 更可靠的
投屏路径可能是 CoreMediaIO/AVFoundation 原生后端，而不是 pyusb 直连 Valeria bulk；
但如果 QuickTime Player 同机选择某台设备也无法显示屏幕，且日志同样是
`StartStream throwing valeria connection not seen`，先按系统/设备临时状态处理：
`imirror reset`、重插线、重启 iPhone 和 Mac。重启后 QuickTime 能恢复时，不应归因到
Python 协议序列化、端点选择或 iOS 18.x 兼容性。

## H 类：QuickTime 配置状态判定混淆

**现象**：
- macOS 执行关闭 QT 后，IORegistry 显示当前活动配置已回到 #5，但 `devices` 仍显示
  “QT配置: 已激活”；随后 `record` 会跳过激活路径。
- QT 配置 #6 同时包含 `Apple USB Multiplexor@1` 和 `Valeria@2`，旧 `_inspect`
  会把 `usbmux_config_index` 误记为 #6。
- 在“QT 描述符已存在、active 仍是 #5”的状态下，再发 vendor `0x40/0x52/index=2`
  不会切回 #6，导致 `activate` 超时；直接 `set_configuration(qt_config)` 可恢复到 #6。

**根因**：把“能读到 QT/Valeria 配置描述符”当成“QT 配置当前已激活”。但 macOS 上
关闭/切回普通配置后，设备仍可能保留 #6 描述符，真实活动配置要用
`get_active_configuration()` 判断。

**修复**（本次提交）：
- `IosDevice` 新增 `active_config_index` 与 `qt_available`，`qt_enabled` 改为
  “active config == qt config”。
- `devices --json` 输出 `active_config` / `usbmux_config` / `qt_config` /
  `qt_available` / `qt_enabled`，便于脚本区分“可用”和“已激活”。
- `_inspect` 只把“有 usbmux 且没有 QT”的配置当普通 mux 配置，避免 QT 配置 #6
  覆盖真实 mux 配置 #5。
- `enable_qt_config` 在 QT 描述符已存在但未激活时，先 `set_configuration(qt_config)`，
  失败才回退到 vendor 激活请求。

**防复发规约**：
- 文档和日志里区分“QT 描述符存在/可用”和“QT 配置当前激活”。
- 恢复普通模式、重新激活、`devices` 展示都必须看 `active_config_index`，不能只看
  `qt_config_index != -1`。
- 提供面向联调的 `imirror reset`：当 macOS 残留 QT 描述符或半激活状态影响判断时，
  先 USB reset 并等待重新枚举，再继续 raw USB 或 macOS 原生路径验证。

## B 类：USB 字符串描述符含不可见字符

**现象**：`--udid 00008110000275943EEB801E`（从 doctor 输出复制的）报"未找到设备"，
但 doctor/devices 又明明列出了这台。

**根因**：iPhone 的 USB 序列号/产品名描述符带 **NUL 填充**（`\x00`），打印时不可见，
但字符串**比较时不相等**。

**防复发规约**：
- 所有从 USB 描述符读来的字符串，一律过 `discovery.clean_str()`（去 NUL + strip）
  再使用/比较。新增读描述符的地方记得套。
- 面向用户的匹配（如 `--udid`）再做一层归一化：去横杠、转小写（见 F 类）。

修复 commit：8574a22

## C 类：Windows USB 平台特性（非 bug，是必须适配的现实）

Windows 的 USB 栈与 Linux/macOS 差异大，以下都不是代码错误，而是**必须处理的平台现实**：

1. **复合设备必须处理整机/父节点**
   - iPhone 是 USB 复合设备，只处理某个子接口**不生效**。
   - 处理：当前路线优先使用 `chotgpt/quicktime_video_hack_windows` 内置驱动安装器，
     按 USB ID `05AC 12A8` 或设备序列号处理整机节点。

2. **一台设备有"两个形态"，各记各的驱动**
   - iPhone 普通形态 vs 激活投屏后的新形态，Windows 视为两个设备实例。
     普通形态准备过驱动后，激活切到新形态，Windows 又派默认驱动 → 需**二次准备驱动**。
   - 处理：激活失败的报错内嵌自救指引，优先重开内置驱动安装器处理新形态。

3. **激活后重枚举慢**
   - Windows 首次给新形态装驱动可能 >5 秒，原轮询 5 秒（10×0.5s）超时。
   - 处理：轮询加长到 15 秒（30×0.5s）。修复 commit：8574a22

4. **`libusb_set_configuration` 返回 NOT_SUPPORTED（Windows 头号硬骨头）**
   - 现象：录制/投屏时报 `set_configuration(5) 失败: Operation not supported`，
     日志显示"活动配置 #1 -> 目标 QT 配置 #5"，最终"活动配置 #1 里没有 QuickTime 接口"。
   - **根因（一度判断错，真机日志纠正）**：iPhone 有多个 USB 配置，投屏接口在
     **非默认配置**（实测 #5，默认活动的是 #1）。必须切到 #5 才能 claim 投屏接口。
     而 **libusbK/WinUSB 这类 WinUSB 家族驱动在 Windows 上不支持切换活动配置**
     （libusb_set_configuration 返回 NOT_SUPPORTED，裸控制请求也不行）。
     ⚠️ 我起初错误假设"激活后 QT 配置本就是活动配置"——那只在 Linux/macOS 成立；
     Windows 重枚举后默认仍选 #1。discovery 能读到 #5 的 0x2A 接口，只因 libusb
     可读取**所有**配置描述符，不代表 #5 是活动配置。
   - **真正解决 = 换对驱动**：Windows 上用 **libusb-win32(libusb0)** 这种完整内核
     驱动，它支持切配置；libusbK/WinUSB 不支持。原版 C++ Windows 项目用的也是
     libusb0/过滤层驱动（见 reference README「安装驱动」节）。手册驱动选择表已改为
     首选 libusb-win32。
   - 代码侧兜底（commit 6a56b3a、5769a09、本次）：`adapter.open` 逐级降级
     （活动配置已是目标则跳过 → set_configuration → 裸控制请求 → 全失败在当前配置
     找接口），找不到接口时报错**明确指向"换 libusb-win32 驱动"**，不再是模糊提示。
   - 命名坑：驱动名叫 "libusb-**win32**" 只是历史项目名，在 64 位 Windows 照常工作。

5. **复合子接口的 bulk 读写全超时（Windows 当前未解决的墙）** ⛔
   - 现象：换 libusb-win32 后，切配置#5/claim接口#2/set_altsetting **全部成功**，
     但 bulk **读 12s 无数据、写 20 字节报 `Operation timed out`**，双向不通；
     设备随后 `No such device`。视频/音频 0 帧。
   - **根因（真机日志 + 参考源码确认）**：QT 接口是 USB **复合设备的子接口**，
     libusb-win32 装在**复合父设备**上时，**控制传输(EP0)能用**（所以切配置/claim/
     altsetting 都成功），但 **bulk 传输路由不到子接口的端点** → 读写全超时。
     这是 Windows 复合设备驱动模型的固有问题，非协议 bug（同样的代码在 Linux/macOS
     上 libusb 原生处理，能正常收发）。
   - **⚠️ 2026-07-10 深挖 C++ 源码后, 以下早期推断被推翻并修正**
     （C++ 采集核心源码其实在仓库里: `ios_line_cast_screen/ios_line_cast_screen/src/QuickTime.cpp`,
     双层嵌套目录+GBK 编码导致初次搜索漏掉; 三路 sub-agent 侦察 + 主 agent 逐行核实）:
     - ❌ 旧推断"usbmuxd 经 lockdown 让 QT 成为默认配置" → **实际**: C++ 版的
       usbmuxd.exe 只需在跑(配对/信任用, `idevicepair pair`), **视频流不经它**;
       激活方式与我们完全相同(控制请求 0x40/0x52/idx=2, QuickTime.cpp:1089)。
     - ❌ 旧推断"驱动必须装到 QT 子接口" → **实际**: C++ 的 libwdi 安装也是按
        VID+PID+序列号选**整机节点**(gcusb.cpp:176-199), 与当前整机节点处理思路一致;
       它也调 usb_set_configuration 切配置(QuickTime.cpp:1078)——和我们一样。
     - ✅ **真正的钥匙 = "唤醒敲门"(QuickTime.cpp:893-905)**: 作者注释原话
       "这句应该是重置命令，不调用可能导致无限读取超时"——读超时(-116)时发
       vendor 控制请求 **(0x40, 0x40, 0x6400, 0x6400)** + **主机主动发 PING**。
       Go 版是被动等设备先 PING(Linux/macOS 可行), Windows 上必须主动敲醒。
   - **修复**: adapter.read_loop 读超时时(仅 win32)自动执行敲门+主动 PING
     (`_kick_device`)；bulk OUT 写入统一带 2s 显式超时，敲门里的主动 PING 失败只按
     DEBUG 探测记录，避免 Windows 端点不通时卡住或刷误导性 ERROR。备选路径
     (若敲门仍不通): libusb-win32 **filter 模式**
     (`install-filter install --device=<hwid>`, 叠加不替换、保苹果驱动,
     bulk 不受 filter 门控, 但 set_configuration 归 usbccgp 所有不可调)。
   - **2026-07-10 新日志修正**：Windows 已能收到设备首个 `PING` 且回复写出成功，
     说明 bulk IN/OUT 初始链路曾经打通；随后 2s 读超时触发 vendor 敲门，反而让主动
     PING 和停止报文进入 `Operation timed out`。修正为：敲门只在尚未收到任何设备数据前
     执行；一旦收到过设备字节，后续读超时只等待/记录，不再发 vendor reset 打断会话。
   - **2026-07-10 复测补充**：失败会话后设备可能残留在 active QT 配置 #5，下一次
     `record` 直接打开接口时不再收到首个 PING。根因是 active QT 只说明 USB 配置正确，
     不代表 Valeria/QuickTime 会话已重新武装。修正为：Windows 的 `record/gui/activate`
     在设备已是 active QT 时也强制重发一次 QT enable，请设备重新进入 PING/SYNC 起点。
   - **2026-07-10 再复测补充**：设备处在 active #1、QT 描述符 #5 已暴露时，Windows
     直接 `set_configuration(5)` 可让 claim/set_altsetting 成功，但仍没有首个 PING，主动
     PING 和停止报文也全超时。根因是直接 SET_CONFIGURATION 只改变 USB active config，
     不一定重新武装 QuickTime/Valeria 会话。修正为：Windows 在 `qt_available=true` 但
     `qt_enabled=false` 时优先重发 Apple QT enable (`0x40/0x52/index=2`)，失败才回退
     `set_configuration(qt_config)`。
   - 上游背书: Go 原作者 README 明言 "I have given up on windows support" ——
     Go 路线在 Windows 就是死路, C++ 的敲门方案是唯一被验证可行的。
    - 附带现象：设备每次重新枚举后 libusb-win32 绑定丢失，需重开内置驱动安装器处理新形态——
      复合/重枚举根因的表现。

**防复发规约**：
- 新增任何 USB 操作前，先想"这在 Windows 的 WinUSB/libusbK 下支持吗"，
  默认给降级路径，不要假设 Linux 的行为在 Windows 成立。
- Windows 上"控制传输能用但 bulk 不通"时，先核对 C++ 参考里的读超时敲门路径：
  vendor `0x40/0x40/0x6400/0x6400` + 主动 PING；不要再误判成 usbmuxd 承载视频流。
- Windows 上"active config 正确但没有首个 PING"时，先按会话未武装处理，重发
  Apple QT enable；不要只看 `set_configuration`/claim/set_altsetting 成功就判定会话已启动。
- Windows 特有的操作现实同步进 `docs/真机联调手册.md` 的排障小节。

## D 类：判定逻辑对复合设备过严

**现象**：换好驱动后 doctor 仍报失败——它要求"所有 Apple USB 节点都可访问"，
但复合设备天然有一批读不了的子接口节点。

**根因**：把"部分节点不可访问"当成了"环境不可用"，一票否决。

**防复发规约**：
- 判定"环境是否就绪"以**能否识别出目标 iOS 设备**为准，不以"所有节点都完美"为准。
  不可访问的节点降级为 ⚠ 提示，不阻断。

修复 commit：7da25fb

## E 类：脚本 / 工具链

   - 处理：手册警告不要用 `uv sync`，装包统一 `uv pip install`；误用后重跑一键脚本恢复。

## F 类：用户输入容错

**现象**：
- `--udid` 传入 usbmuxd 风格带横杠的 `00008110-000275943EEB801E` 报未找到。
- 多台 iOS 设备同时可见时，未传 `--udid` 默认选第一台，可能把激活/录制跑到错误设备；
  激活失败时还会暴露 Python traceback。

**根因**：USB 序列号不带横杠，usbmuxd/Xcode 显示的 udid 带横杠，两者格式不一；
多设备场景下“默认第一台”不可预测，面向用户的 CLI 不应让 RuntimeError 穿透成 traceback。

**防复发规约**：
- 面向用户的标识匹配做归一化（去横杠 + 转小写）后比较，两种写法都接受。
- 匹配失败时，报错里**列出当前可见的合法值**，让用户直接照抄。
- 多台设备且未显式 `--udid` 时直接停止并列出候选，不默认选第一台。
- CLI 顶层捕获 `RuntimeError` 并打印友好错误，保留 `-v` 日志用于定位，不向用户抛栈。

修复 commit：6a56b3a；本次提交
